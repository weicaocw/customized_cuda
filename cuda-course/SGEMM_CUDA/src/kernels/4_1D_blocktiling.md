# GPU 物理层级全解（建立心智模型）

在进入微观旅行前，我们必须在你的脑海中建起这座“GPU 超级加工厂”。按照从宏观到微观、从慢到快的顺序：

## 1. 内存层级（仓库与工作台）
* **RAM（系统内存）：** 插在电脑主板上的内存条（CPU 的地盘）。GPU 想要这里的数据，必须通过 PCIe 总线（跨海大桥）极其缓慢地运过来。
* **VRAM（显存 / Global Memory）：** GPU 自己的大仓库（比如你买显卡常说的 12GB、24GB）。它是所有的起点和终点，容量大，但距离计算核心很远，**速度极慢**。
* **L2 Cache（二级缓存）：** 位于 GPU 芯片中心的中转站。所有车间（SM）共享。当你从 VRAM 读数据时，数据会先缓存在这里。
* **L1 Cache / SMEM（一级缓存 / 共享内存）：** 它们物理上是**同一块极其昂贵的硅片（SRAM）**，建在每一个加工车间（SM）的内部，**速度极快**。
    * *L1 是自动挡：* 硬件自动帮你缓存。
    * *SMEM 是手动挡：* 就是代码里的 `__shared__`，你写代码强制规定这里放什么。
* **Registers（寄存器）：** 缝在每个工人（Thread）工作服上的口袋。容量极其微小（几百个字节），但它是唯一能直接喂给计算机器（ALU）的数据源。**速度是光速（零延迟）。**

## 2. 计算层级（车间与工人）
* **SM（Streaming Multiprocessor，流式多处理器）：** 独立的加工车间。一张显卡通常有几十到上百个 SM。每个 SM 里有自己的 SMEM、寄存器堆和计算核心。
* **Warp（线程束）：** **这是 GPU 物理执行的绝对核心概念！** 32 个工人被铁链绑在一起，组成一个 Warp。他们共用一个指挥官，**在任何一个时钟周期，这 32 个人必须执行完全相同的动作**（比如同时去读内存，同时做加法）。
* **Thread（线程）：** 最底层的单个工人。负责算具体的数字。

## 3. 高能微观旅行 —— 一段代码的物理狂飙

现在，我们将把刚才的 `sgemm1DBlocktiling` 扔进这座物理工厂。
假设我们的 Block（512 个工人，即 16 个 Warp）被分配到了第 7 号 SM 车间。

### 幕布拉开：搬砖阶段 (Populate SMEM)
```cpp
As[innerRowA * BK + innerColA] = A[innerRowA * K + innerColA];
```
1. **指挥官下令：** 0 号 Warp 的 32 个工人走向 VRAM 仓库，准备搬 $A$ 的前 32 个元素。
2. **合并访存的物理奇迹：** 因为我们的坐标映射极其精妙，这 32 个人要搬的物理地址是**完美紧挨着的**！
3. **L2 缓存的放行：** 显存控制器一看，爽快地发出一条 128 Bytes 的超级总线请求，一次性把这 32 个 `float` 从 VRAM 抽出，途经 L2 Cache，直接拍在了 7 号车间的共享内存（SMEM `As`）上。
4. 16 个 Warp 轮番上阵，瞬间把 512 个 $A$ 和 512 个 $B$ 搬完了。

### 高潮降临：计算阶段 (The Compute Loop)
```cpp
float tmpB = Bs[dotIdx * BN + threadCol];
```
1. **共享内存的“广播机制 (Broadcast)”：**
   现在，Warp 0 的 32 个人准备去 SMEM 拿 $B$ 矩阵的数据。
   在当前的 `dotIdx` 下，由于 `threadCol` 对于相邻线程是连续递增的，这 32 个人读取的是 SMEM 中连续的 32 个地址。SMEM 内部有 32 个 Bank（通道），这 32 个通道同时开火，毫无冲突（Bank Conflict-Free），**只需 1 个时钟周期**，就顺滑地把数据塞进了 32 个人的寄存器 `tmpB` 里。

```cpp
threadResults[resIdx] += As[(threadRow * TM + resIdx) * BK + dotIdx] * tmpB;
```
1. **寄存器的狂舞：**
   进入内层循环 8 次（`TM=8`）。
   请注意：这 8 次循环中，`tmpB` **永远在寄存器里！不需要去任何地方读！**
   工人每次只需去 SMEM 读一个 `As`，然后直接把口袋里的 `tmpB` 和 `As` 丢进自己的算术逻辑单元（ALU）中。
2. **FFMA 指令的咆哮：**
   ALU 执行的是硬件级的 Fused Multiply-Add（融合乘加指令），在一个时钟周期内，同时完成乘法和累加，并将结果稳稳地存回另一个寄存器 `threadResults` 中。
3. **隐藏延迟的艺术：**
   由于这里没有任何 `if` 分支，也没有复杂的寻址，编译出来的汇编代码全是极其密集的 `FFMA` 指令。此时的 SM 车间，计算单元的吞吐量被推到了物理极限（接近 100% 的 Compute Bound）。

### 完美落幕：
当所有的循环结束，512 个工人各自口袋里揣着 8 个滚烫的最终结果，走向 VRAM，将它们整齐划一地写回矩阵 $C$。

## 导师结语
看到了吗？这段代码之所以快，是因为它在每一步都极其“谄媚”地迎合了 GPU 的物理脾气：
* 迎合 VRAM：用 1D 坐标实现 100% 合并访存。
* 迎合 SMEM：用巧妙的步进避免了 Bank 冲突。
* 迎合 Registers：用 `TM` 列计算将复用的 $B$ 数据死死锁在寄存器中，极大降低了对 SMEM 的访问压力。

这就是异构计算的最高浪漫 —— 用纯粹的软件逻辑，完美地支配了上百亿个物理晶体管的跳动！
